Glosario de Calidad y Desarrollo de Software

Recursos y Aprendizaje: Calidad y Desarrollo de Software

  • QA (Quality Assurance): proceso sistemático para garantizar la calidad del software durante su desarrollo.
  • Testing: actividad que evalúa si el software cumple con los requisitos especificados.
  • Bug / Defecto: error en el software que causa resultados incorrectos o fallos.
  • CI/CD: integración y entrega continua; automatiza la compilación, prueba y despliegue del software.
  • Refactorización: mejorar el código existente sin cambiar su comportamiento.
  • Revisión de código (Code Review): evaluación del código por parte de otros desarrolladores para asegurar calidad y buenas prácticas.
  • Unit Testing (Pruebas Unitarias): pruebas que verifican el comportamiento de pequeñas partes individuales del código.
  • TDD (Test Driven Development): desarrollo guiado por pruebas: primero se escriben las pruebas, luego el código.
  • Métricas de Calidad: indicadores que ayudan a medir aspectos como cobertura de pruebas, complejidad o cantidad de defectos.
  • DevOps: cultura que integra desarrollo y operaciones para mejorar la entrega continua y automatizada del software.

Preguntas Frecuentes

¿Cuál es la diferencia entre QA y Testing? Ir a la sección relacionada ↗
QA es un proceso más amplio que incluye todas las actividades para asegurar la calidad. El testing es parte de QA y se enfoca en detectar errores.
¿Por qué es importante la calidad del software? Ir a la sección relacionada ↗
Reduce errores en producción, mejora la experiencia del usuario y ahorra costos a largo plazo.
¿Cuándo se debe empezar a probar el software? Ir a la sección relacionada ↗
Desde las primeras etapas del desarrollo para detectar errores antes y a menor costo.
¿Qué herramientas se usan para pruebas automatizadas? Ir a la sección relacionada ↗
Selenium, JUnit, PyTest, Postman, Jest, entre otras.
¿Qué es un caso de prueba? Ir a la sección relacionada ↗
Descripción de lo que se va a probar, con pasos, datos de entrada y resultado esperado.
¿Diferencia entre pruebas funcionales y no funcionales? Ir a la sección relacionada ↗
Funcionales: qué hace el sistema. No funcionales: cómo lo hace (rendimiento, seguridad, usabilidad).

Ejercicios y Casos Prácticos

Ejercicio 1 · Scrum

Contexto: El cliente quiere lanzar “wishlist” y “recordatorio de stock”.

  • Como PO, prioriza 6 historias (valor/risgo/dep.) y define un Sprint Goal claro.
  • Como equipo, desglosa 3 historias en tareas técnicas (8–12h c/u), estima y prepara el Sprint Backlog.
  • Define una DoD corta (tests, review, CI verde, doc mínima).

Entregables: Sprint Goal, Sprint Backlog (tabla), DoD (lista) y plan de demo.

Ejercicio 2 · Kanban

Contexto: Equipo de soporte con tiempos de ciclo altos e impredecibles.

  • Dibuja un tablero con columnas reales y define políticas de entrada/salida.
  • Propón límites WIP por columna y una clase de servicio “Expedite”.
  • Plantea cómo medirás Lead/Cycle Time, Throughput y Aging WIP, y cómo interpretar un CFD.

Entregables: tablero con WIP, políticas explícitas y plan de métricas.

Ejercicio 3 · Cascada / V

Contexto: App médica para registrar signos vitales.

  • Elabora una matriz V mapeando cada fase con su verificación.
  • Define 5 casos de prueba clave (funcionales y de seguridad).
  • Especifica criterios de aceptación y un plan de trazabilidad.

Entregables: diagrama V simple, matriz de trazas y lista de casos de prueba.

Ejercicio 4 · XP / TDD

Contexto: Implementar un carrito con descuentos por volumen.

  • Escribe primero 5 pruebas unitarias (TDD) para: agregar, quitar, totalizar, descuento y borde (0 items).
  • Implementa lo mínimo para pasar cada prueba; refactoriza al final.
  • Configura CI y agrega code review y pair programming como prácticas XP.

Entregables: suite de tests, código refactorizado, README con pasos.
